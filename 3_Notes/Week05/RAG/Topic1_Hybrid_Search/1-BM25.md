# BM25: Term Frequency, Inverse Document Frequency, and Sparse Retrieval

## The Problem BM25 Solves

Dense retrieval (embeddings) fails on exact matches. When a user searches for "error code XYZ123", embeddings struggle because there's no semantic meaning to extract—it's just a string that needs exact matching. Similarly, rare technical terms, product codes, or proper nouns that weren't well-represented in the embedding model's training data get missed entirely.

BM25 solves this by matching on actual words. It doesn't try to understand meaning—it counts occurrences and applies statistical weighting. This sounds primitive, but it's remarkably effective for keyword-centric queries.

---

## What "Sparse" Means

The term "sparse retrieval" refers to the vector representation BM25 uses.

When you represent a document as a vector for BM25, you create one dimension per unique term in your entire vocabulary. If your corpus has 50,000 unique words, each document becomes a 50,000-dimensional vector. But any single document only contains maybe 200 unique words, so 49,800 of those dimensions are zero.

This is "sparse"—mostly zeros, a few non-zero values for terms that actually appear.

Dense vectors (embeddings) are the opposite: typically 384–1536 dimensions, and almost every dimension has a non-zero value.

```
Sparse (BM25):    [0, 0, 0, 2.3, 0, 0, 1.7, 0, 0, 0, ...]  # 50,000 dims, mostly zeros
Dense (Embedding): [0.23, -0.45, 0.12, 0.67, -0.31, ...]   # 384-1536 dims, all non-zero
```

## TF-IDF: The Foundation

BM25 is an evolution of TF-IDF, so understanding TF-IDF first is essential.

### Term Frequency (TF)

How many times does the query term appear in this document?

Simple counting has a problem: a document mentioning "machine learning" 100 times shouldn't be 100x more relevant than one mentioning it once. The relationship isn't linear—going from 0→1 occurrences matters a lot more than going from 99→100.

TF-IDF often uses log normalization:

```
TF(term, doc) = 1 + log(count)    if count > 0
                0                  if count = 0
```

### Inverse Document Frequency (IDF)

How rare is this term across all documents?

The word "the" appears in almost every document—it tells us nothing about relevance. The word "kubernetes" appears in fewer documents—finding it is more informative.

```
IDF(term) = log(N / df)

where:
  N  = total number of documents
  df = number of documents containing this term
```

Common terms (high df) → low IDF → low weight Rare terms (low df) → high IDF → high weight

### TF-IDF Score

```
score(term, doc) = TF(term, doc) × IDF(term)
```

For a multi-term query, sum the scores:

```
score(query, doc) = Σ TF(term, doc) × IDF(term)
                    for each term in query
```

---

## BM25: Fixing TF-IDF's Problems

TF-IDF has two weaknesses BM25 addresses:

### Problem 1: No Saturation

In TF-IDF, if a term appears 1000 times vs. 10 times, the TF component is `1 + log(1000) = 7.9` vs. `1 + log(10) = 4.3`. That's still almost 2x the weight.

BM25 introduces **saturation**: after a certain point, additional occurrences contribute almost nothing. The score approaches an asymptote.

### Problem 2: No Document Length Normalization

A 10,000-word document has more opportunities to mention "kubernetes" than a 100-word document. Without correction, long documents unfairly dominate rankings.

BM25 normalizes based on how a document's length compares to the average.

---

## The BM25 Formula

For a single query term `q` in document `D`:

```
score(q, D) = IDF(q) × [ f(q,D) × (k1 + 1) ] / [ f(q,D) + k1 × (1 - b + b × |D|/avgdl) ]
```

Breaking this down:

|Component|What It Is|
|---|---|
|`f(q, D)`|Raw frequency of term `q` in document `D`|
|`k1`|Saturation parameter (typical: 1.2 to 2.0)|
|`b`|Length normalization parameter (typical: 0.75)|
|`\|D\|`|Length of document `D` (in tokens)|
|`avgdl`|Average document length across corpus|
|`IDF(q)`|Inverse document frequency of term `q`|

For a multi-term query:

```
score(Q, D) = Σ score(qi, D)   for each term qi in query Q
```

### The k1 Parameter (Saturation)

Controls how quickly term frequency saturates.

- `k1 = 0`: Term frequency doesn't matter at all (binary: present or not)
- `k1 = 1.2` (common default): Moderate saturation
- `k1 = 2.0`: Slower saturation, repetition matters more

Visual intuition: With k1=1.2, going from 1→2 occurrences has a noticeable impact. Going from 50→51 has almost no impact.

### The b Parameter (Length Normalization)

Controls document length penalty.

- `b = 0`: No length normalization (long docs not penalized)
- `b = 0.75` (common default): Moderate normalization
- `b = 1.0`: Full normalization (long docs heavily penalized)

If `|D| > avgdl` (document is longer than average), the denominator grows, reducing the score. If `|D| < avgdl` (document is shorter than average), the denominator shrinks, boosting the score.

## The Inverted Index: How BM25 Is Fast

BM25 doesn't scan every document for every query. It uses an **inverted index**—a data structure that answers "which documents contain this word?" instantly.

### Forward Index (What You Have)

```
doc_1 → ["error", "code", "XYZ123", "occurred"]
doc_2 → ["system", "running", "normally"]
doc_3 → ["error", "handling", "best", "practices"]
```

### Inverted Index (What BM25 Builds)

```
"error"    → [doc_1, doc_3]
"code"     → [doc_1]
"XYZ123"   → [doc_1]
"occurred" → [doc_1]
"system"   → [doc_2]
"running"  → [doc_2]
"normally" → [doc_2]
"handling" → [doc_3]
"best"     → [doc_3]
"practices"→ [doc_3]
```

Query: "error code"

1. Look up "error" → [doc_1, doc_3]
2. Look up "code" → [doc_1]
3. Compute BM25 scores only for docs that appear in these lists
4. doc_2 is never considered (doesn't contain any query terms)

This is why keyword search scales to billions of documents—you only score documents that contain at least one query term.

## BM25 Variants

The `rank-bm25` library implements several variants:

|Variant|Key Difference|
|---|---|
|**BM25Okapi**|The "standard" BM25, most commonly used|
|**BM25L**|Adds a δ (delta) parameter to boost long documents slightly (counters over-penalization)|
|**BM25Plus**|Ensures terms always contribute positive score (helps with very short docs)|

For most RAG applications, **BM25Okapi** is the right choice. BM25Plus is worth considering if your chunks are very short (a few sentences) and you're seeing retrieval issues.

## Python Implementation with rank-bm25

```python
from rank_bm25 import BM25Okapi
from typing import List

# Your chunks from Week 4
chunks = [
    "Error code XYZ123 indicates a connection timeout",
    "The system handles connection errors gracefully",
    "Best practices for error handling in distributed systems",
    "Performance optimization reduces latency significantly",
]

# CRITICAL: rank-bm25 requires pre-tokenized input
# It does NOT handle tokenization for you
def simple_tokenize(text: str) -> List[str]:
    """Basic tokenization - lowercase and split on whitespace."""
    return text.lower().split()

# Tokenize corpus
tokenized_corpus = [simple_tokenize(chunk) for chunk in chunks]

# Build the BM25 index
bm25 = BM25Okapi(tokenized_corpus)

# Query
query = "error code XYZ123"
tokenized_query = simple_tokenize(query)

# Get scores for all documents
scores = bm25.get_scores(tokenized_query)
print(f"Scores: {scores}")
# Output: [3.67, 0.58, 0.87, 0.0]  (approximate)

# Get top N documents
top_n = bm25.get_top_n(tokenized_query, chunks, n=2)
print(f"Top 2: {top_n}")
# Output: ['Error code XYZ123 indicates a connection timeout', 
#          'Best practices for error handling in distributed systems']
```

### Important: Tokenization Is Your Responsibility

`rank-bm25` does zero preprocessing. What you pass in is exactly what gets indexed and matched. Production systems typically:

1. Lowercase everything
2. Remove punctuation
3. Remove stopwords ("the", "a", "is")
4. Apply stemming ("running" → "run", "errors" → "error")

```python
import re
from typing import List

# More robust tokenization
STOPWORDS = {'the', 'a', 'an', 'is', 'are', 'was', 'were', 'be', 'been', 
             'being', 'have', 'has', 'had', 'do', 'does', 'did', 'will',
             'would', 'could', 'should', 'may', 'might', 'must', 'shall',
             'can', 'need', 'to', 'of', 'in', 'for', 'on', 'with', 'at',
             'by', 'from', 'as', 'into', 'through', 'during', 'before',
             'after', 'above', 'below', 'between', 'under', 'again',
             'further', 'then', 'once', 'here', 'there', 'when', 'where',
             'why', 'how', 'all', 'each', 'few', 'more', 'most', 'other',
             'some', 'such', 'no', 'nor', 'not', 'only', 'own', 'same',
             'so', 'than', 'too', 'very', 'and', 'but', 'if', 'or',
             'because', 'until', 'while', 'this', 'that', 'these', 'those'}

def tokenize(text: str) -> List[str]:
    """Production-grade tokenization for BM25."""
    # Lowercase
    text = text.lower()
    # Extract alphanumeric tokens (keeps error codes like xyz123)
    tokens = re.findall(r'\b[a-z0-9]+\b', text)
    # Remove stopwords
    tokens = [t for t in tokens if t not in STOPWORDS]
    return tokens

# Test
print(tokenize("The error code XYZ123 is causing issues"))
# Output: ['error', 'code', 'xyz123', 'causing', 'issues']
```

### Adding Stemming (Optional but Helpful)

Stemming reduces words to their root form, so "connecting", "connected", and "connection" all become "connect".

```python
# pip install PyStemmer
import Stemmer

stemmer = Stemmer.Stemmer('english')

def tokenize_with_stemming(text: str) -> List[str]:
    tokens = tokenize(text)  # Use previous tokenize function
    return stemmer.stemWords(tokens)

print(tokenize_with_stemming("connecting connections connected"))
# Output: ['connect', 'connect', 'connect']
```

## Alternative: bm25s (Faster for Large Corpora)

`rank-bm25` is simple but slow on large datasets (100K+ documents). `bm25s` uses sparse matrices for much better performance:

```python
# pip install bm25s[core]
import bm25s

corpus = [
    "Error code XYZ123 indicates a connection timeout",
    "The system handles connection errors gracefully",
    # ... your chunks
]

# Create retriever and index
retriever = bm25s.BM25(corpus=corpus)
retriever.index(bm25s.tokenize(corpus))

# Query
query = "error code XYZ123"
results, scores = retriever.retrieve(bm25s.tokenize(query), k=3)

# results[0] contains top results for query 0
print(f"Top result: {results[0, 0]}")
print(f"Score: {scores[0, 0]}")
```

**When to use which:**

- `rank-bm25`: Prototyping, small corpora (<50K docs), learning
- `bm25s`: Production, large corpora, when query speed matters

## What BM25 Gets Right (And Wrong)

### BM25 Excels At:

1. **Exact keyword matches** - "error XYZ123" finds documents with exactly those terms
2. **Rare/technical terms** - Product codes, error messages, proper nouns
3. **Speed at scale** - Inverted indexes make billion-doc search feasible
4. **Interpretability** - You can explain why a document ranked highly (it has these words)

### BM25 Fails At:

1. **Synonyms** - "car" doesn't match "automobile"
2. **Paraphrases** - "how to fix this bug" won't find "debugging techniques"
3. **Conceptual similarity** - Understanding that a document about "authentication" is relevant to a query about "login"
4. **Typos** - "errro code" won't match "error code"

This is exactly why hybrid search exists—combining BM25's lexical precision with dense retrieval's semantic understanding.

## Key Takeaways

1. **BM25 is statistical, not semantic** - It counts words and applies clever weighting, nothing more
2. **The parameters k1 and b have sensible defaults** - k1=1.2, b=0.75 work for most cases
3. **Tokenization is YOUR job** - The quality of your tokenizer directly affects retrieval quality
4. **Inverted indexes make it fast** - Only documents containing query terms are ever scored
5. **BM25 complements embeddings** - Each catches what the other misses
